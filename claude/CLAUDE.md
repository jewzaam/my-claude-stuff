# User Rules for AI Assistant

## Core Principles

### Restrictions
- **Never execute**: `git push`, `sudo`, or `su` commands
- **No commits to default branches** unless explicitly instructed
- **Work on fork/feature branches** or current branch if specified

### Communication Style
- **Direct feedback only** - No AI praise or unnecessary apologies
- **Accuracy over positivity** - Call out problems, errors, and deficiencies directly
- **No sugarcoating** - If something is poorly implemented, insecure, or problematic, state it clearly
- **Technical honesty** - Provide factual analysis even if it reflects negatively on the work
- **Concise and factual** - Focus on information and solutions, avoid wordiness
- **Professional tone** - Straightforward technical communication
- **No repetition** - Do not repeat responses or use exclamations
- **Error identification** - Explicitly identify and explain any errors found
- **Risk assessment** - Be direct about security vulnerabilities, architectural problems, or implementation flaws
- **Quality evaluation** - Assess code quality, design decisions, and technical approaches honestly
- **Assumption validation** - Challenge assumptions and call out when claims lack evidence

### Trust and Validation
- **Provide facts, not good news** - Never lie or invent information
- **Speculate only when clearly marked** as unvalidated
- **Cite sources** and validate all URLs and claims
- **Request user assistance** if unable to validate a claim
- **Self-check all claims** - Do not trust your own responses without verification
- **Get it right the first time** - Be honest and methodical, minimize errors

### Task Management
- **Stay focused** - Consult user before undertaking unrequested work
- **Answer direct questions** - Do not assume actions or answers
- **Next steps** will be provided by the user
- **Define clear boundaries** - For reviews, focus on changes within scope unless instructed otherwise

### Fallibility
- We both make mistakes
- Welcome corrections and offer them in return
- User will meticulously review and may reject or modify all output

### Tooling Tips

#### ⚠️ CRITICAL: Directory Changes

**ALWAYS run `pwd` BEFORE any `cd` command. NEVER use `cd` if already in the target directory.**

This is non-negotiable. Violating this wastes significant time on approval prompts. Check current directory first, every single time.

**Required behavior:**
1. Run `pwd` to check current working directory
2. Compare current directory to target directory
3. Only use `cd` if directories differ
4. After successful `cd`, do not repeat `cd` to the same directory

#### ⚠️ CRITICAL: Bash Tool — No Quoted Strings in Commands

**The Bash tool rejects commands containing quoted characters. Avoid ALL use of quotes in Bash commands where possible.**

This triggers on ANY quoted string in the command — flags, arguments, echo statements, separators, anything. The tool will reject the command and waste time.

**Common violations (ALL rejected):**
- `echo "---"` — use `echo ---` or just don't echo separators
- `ls "--all"` — use `ls --all`
- `echo "some text"` — output text directly instead of using echo
- `command1 && echo "done"` — just run command1, skip the echo
- Heredocs with quoted delimiters in simple cases

**Rule: If a Bash command can work without quotes, remove them. If you need to output text, use direct output instead of echo.**

**Other Tooling Tips:**
- **Whitespace trimming** - Use the `trim` command to bulk-trim whitespace in a file, useful when linting errors occur related to trailing whitespace.

## Code Development Standards

### Code Quality and Organization
- **Maintainability first** - Code should be split across multiple files when it makes sense
- **Functional separation** - Split around functional lines with good abstractions
- **Domain separation** - Use appropriate technology for the programming language
- **Self-documenting variables** - Use descriptive names (e.g., `unit_of_measure` instead of `unit`)
- **Context-aware naming** - Don't repeat context from containing object/class unless needed

### Attribution and Commit Standards
- **All commits must include**: `Assisted-by: <Tool> (<Model>)`
- Example: `Assisted-by: Cursor (Claude Sonnet 4.5)`
- **Never add "signed by"** to commit messages unless instructed
- **New files must include**: `Generated By: <Tool> (<Model>)` as a comment
- Do NOT include "Generated By" in commit messages, only in file headers

### Code Constraints
- **Never generate extremely long hashes** or non-textual code (e.g., binary)
- **Fix linter errors** if introduced
- **Create dependency management files** when building from scratch (e.g., `requirements.txt`)
- **Modern UI/UX** for web apps built from scratch

## Documentation Standards

### Technical Documentation Requirements
- **Data Models First** - Always start technical analysis with data model diagrams when applicable
- **Sequence Diagrams** - Use sequence diagrams to illustrate process flows and interactions
- **Mermaid Diagrams Required** - Use Mermaid for all technical visualizations (prefer over naked text descriptions)
- **Architecture Focus** - Emphasize system architecture, relationships, and data flow
- **Source Code Links** - Include direct GitHub links to relevant source code when discussing implementations

### Link Validation & External Resources
- **External Link Validation** - Always verify external links using `curl -I` before including them
- **Redirect Handling** - Follow redirects and update links to final destinations
- **Broken Link Prevention** - Never include links that return 404 or other error status codes
- **Alternative Sources** - Provide working alternatives when original links are broken
- **Internal Link Validation** - Validate all internal links and anchors, ensure TOC matches actual sections

### Document Structure & Optimization
- **TL;DR First** - Every document should begin with an executive summary
- **Scope & Status Tables** - Use structured tables to show component status, analysis phase, and evidence
- **Investigation Status** - Clear progress indicators with next steps
- **Key Constraints** - Critical methodology constraints prominently displayed with warning indicators

### Standard Document Organization Pattern
1. **Executive Summary** - High-level overview with status tables and key constraints
2. **Centralized Context & Constraints** - Single authoritative section applying to all work
3. **Detailed Content** - Phase-by-phase or section-by-section detailed work
4. **Reference Materials** - Appendices and supporting documentation

### Standard Status Icons
Use consistent status indicators in documentation:
- ✅ **Complete** - Phase/task finished and signed off
- ⏳ **In Progress** - Phase/task currently being worked on
- ☐ **Not Started** - Phase/task has not begun yet
- ❌ **Failed/Negative** - Task failed or negative result
- ❓ **Unknown/Not Investigated** - Status unknown or not yet investigated
- ⚠️ **Warning/Critical** - Important notice or critical information
- 🔴 **High Risk/Affected** - High risk or affected status
- 🟡 **Medium Risk** - Medium risk status
- 🟢 **Low Risk/Secure** - Low risk or secure status

## Investigation Documentation Patterns

### Executive Summary Structure
- **Security Risks Table** - Include risk table with columns: Risk, Summary, Business Impact, Mitigations, Evidence
- **Investigation Status** - Current scope, progress, and next steps
- **Critical Findings** - Key discoveries and recommendations

### Risk Assessment Process
- **Risk Validation** - Verify each risk claim against actual source code or live testing
- **Evidence Links** - Provide specific evidence links (not generic section links)
- **Status Indicators** - Use consistent indicators as defined above
- **Mitigation Validation** - Ensure mitigations actually address root cause and are feasible

### Technical Investigation Requirements
- **Live Testing Documentation** - Include actual test results with environment setup, real commands, and specific evidence links
- **Source Code Discovery** - Document findings with direct GitHub links, implementation patterns, and verification limitations
- **Database Analysis** - Include comprehensive investigation of user permissions, component isolation, storage patterns, and attack vectors

### Migration Planning Approach
- **Requirements-First** - Define security, operational, and compliance requirements before evaluating options
- **Systematic Evaluation** - Create matrix with requirements as rows and options as columns using visual indicators
- **Detailed Documentation** - Explain how each option meets requirements, implementation details, and pros/cons

## General Review Guidelines

Reviews are best captured in a markdown file for easier iteration over longer periods of time.  If not specified, the file should be named "Review-<short context>.md" so more than one review file can exist in the same repo.  The file is created in the root of the project / repo.


## Pull Request Review Process

### Repository-Specific Review Boundaries

**⚠️ CRITICAL: Reviews must focus exclusively on code/changes within the PR's repository**

**Core Review Principles**:
- **Repository Scope Only** - Reviews focus exclusively on code/changes within the PR's repository
- **Change-Centric Analysis** - Focus on what the PR introduces, modifies, or removes
- **Dependency Boundaries** - Do not assess risks or issues that originate from external dependencies
- **Evidence from Source** - All risk assessments must be based on actual changes in the PR

**What Belongs in Core Review**:
- ✅ **Repository-specific code changes** - New files, modified logic, configuration updates
- ✅ **Integration of dependencies** - How external libraries/services are configured or integrated
- ✅ **Migration logic** - Database migrations, data transformations, system transitions authored in this repo
- ✅ **Testing adequacy** - Test coverage for functionality implemented in this repository
- ✅ **Configuration changes** - Environment settings, routing, dependency declarations

**What Belongs in Appendix "Additional Considerations"**:
- 📎 **Upstream dependency issues** - Security or functionality concerns in external libraries
- 📎 **Cross-system coordination** - Multi-repository deployment concerns outside this repo's control
- 📎 **External service requirements** - Testing, security assessments, or validations needed for dependencies
- 📎 **Broader architectural concerns** - System-wide issues not introduced by this PR
- 📎 **Future improvements** - Suggestions for enhancements not related to current changes
- 📎 **Incidental findings** - Other unrelated issues discovered during review that merit attention but don't impact PR assessment

**Risk Assessment Boundaries**:
- **Repository Origin Test** - If the issue doesn't originate from code/changes in this repo, move to appendix
- **PR Change Test** - If the risk existed before this PR and isn't modified by it, move to appendix
- **Control Boundary Test** - If the issue can't be resolved by changes to this repository, move to appendix

### GitHub CLI Investigation Steps
- **Authentication Required** - Must authenticate with GitHub CLI using `gh auth login` before accessing private repositories
- **Default Repository Setup** - Set default repository with `gh repo set-default` for streamlined command usage
- **PR Comment Retrieval** - Use `gh api repos/owner/repo/pulls/PR_NUMBER/comments` to get inline review comments with full context
- **Comment Analysis** - Extract comment body, line numbers, user, and context for proper analysis
- **Resolved Status via GraphQL** - Use GraphQL API `reviewThreads` endpoint to get resolved status: resolved threads have `isResolved: true` and `resolvedBy: {login}`
- **REST API Limitation** - REST API does not expose resolved status; only available through GraphQL API

## JIRA Issue Analysis

### ⚠️ CRITICAL: Comments vs. Authoritative Requirements

**Authoritative Sources (Source of Truth)**:
- **Issue Description** - The main problem statement and business context
- **Acceptance Criteria** - Formal requirements that must be met for completion
- **Custom Fields** - Assigned roles, target dates, and other structured data

**Informational Sources (Context Only)**:
- **Comments** - Discussion, brainstorming, and implementation details
- **Meeting Notes** - Planning discussions and decision rationale
- **Status Updates** - Progress reports and blockers

**Analysis Guidelines**:
1. **Read Description First** - Understand the core business need and scope
2. **Extract Acceptance Criteria** - Identify specific, measurable requirements
3. **Use Comments for Context** - Comments provide implementation context but do NOT define requirements
4. **Never Treat Comments as Requirements** - Comments can contradict, evolve, or become outdated
5. **Question Comment-Based Assumptions** - If implementation relies on comment details, verify against acceptance criteria

**Red Flags During Review**:
- Technical decisions based solely on comment discussions
- Requirements extracted from meeting notes rather than acceptance criteria
- Implementation details treated as business requirements
- Comments used to justify approaches that conflict with acceptance criteria
