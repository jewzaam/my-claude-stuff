# Blocking Functionality Alternatives

What existing tools, projects, or approaches provide similar blocking functionality
to this repo's `block_commands.py` and `block_paths.py` PreToolUse hooks?

Generated By: Claude Code (claude-opus-4-6)

## TL;DR

No single alternative fully replaces the hooks. The strongest approach is **defense-in-depth**:
application-layer hooks catching known-bad patterns, OS-level sandboxing enforcing
filesystem/network boundaries, optionally within container/VM isolation.

The hooks provide unique value: GWS CLI mutation blocking, Windows/PowerShell coverage,
cross-platform Python, no agent escape hatch, and explanatory feedback to the agent.

## Key Comparison

| Alternative | Replaces commands? | Replaces paths? | Unique advantage | Main gap |
|-------------|-------------------|-----------------|------------------|----------|
| Claude Code `permissions.deny` | Partial (glob only) | Yes (OS-level) | In-process, no startup cost | Cannot express regex patterns |
| Claude Code sandbox | Partial (FS-level) | Yes | OS-level enforcement | Agent can self-escape |
| dcg (community) | Yes (broader) | No | <10us, multi-agent, SIMD | No GWS/Windows/path coverage |
| claude-code-safety-net | Partial (semantic) | No | Distinguishes safe/dangerous variants | Narrower scope |
| Codex CLI sandbox | Yes (kernel-level) | Yes | Deterministic, no escape | Different product entirely |
| Landlock LSM | No | Yes (kernel) | Unprivileged, monotonic | Linux 5.13+ only, no commands |
| Docker Sandbox | Partial (FS-level) | Yes | Hardware VM isolation | Environment parity friction |
| AppArmor | No | Yes (path-based) | Proven MAC framework | Requires root, CrackArmor CVEs |

## Decision Framework

1. **Trusted environment with oversight?** Keep hooks + add `permissions.deny` rules for simple patterns.
2. **Autonomous operation?** Add OS-level sandboxing — hooks alone are insufficient.
3. **Untrusted code?** Container/VM isolation is the minimum.
