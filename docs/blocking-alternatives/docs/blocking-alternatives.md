# Blocking Functionality Alternatives: Cited Research

A comparison of approaches for blocking destructive commands and sensitive file access
in AI coding agents, evaluated against this repo's custom PreToolUse hooks.

Generated By: Claude Code (claude-opus-4-6)

---

## Executive Summary

This repo implements two Claude Code PreToolUse hooks:
- **block_commands.py** — regex-based command blocking (git destructive ops, sudo, rm -rf, Windows/PowerShell, GWS CLI mutations)
- **block_paths.py** — blocks access to sensitive directories (~/.ssh, ~/.aws, ~/.kube, ~/.ocm) and credential files

We evaluated five categories of alternatives: Claude Code's native features, community hook projects, other AI agent guardrails, container/sandbox isolation, and OS-level security tools. The research covered 73 sources across official documentation, GitHub projects, vulnerability disclosures, and security research.

### Key Finding

No single alternative fully replaces the repo's hooks. The strongest approach is **defense-in-depth**: application-layer hooks catching known-bad patterns early, combined with OS-level sandboxing enforcing filesystem/network boundaries, optionally within container/VM isolation for blast radius containment.

### Decision Framework

1. **Are you running in a trusted environment with oversight?** The repo's hooks plus Claude Code's native `permissions.deny` rules provide adequate protection.
2. **Are you running autonomously (--dangerously-skip-permissions)?** Add OS-level sandboxing (Claude Code's native sandbox or Docker Sandbox) — hooks alone are insufficient for unattended operation.
3. **Are you running untrusted code?** Container/VM isolation (Docker Sandbox microVMs, Firecracker, devcontainer) is the minimum — application-layer hooks cannot contain a determined adversary.

## Dimension 1: Claude Code Native Features

Full reference: [reference/claude-code-native.md](reference/claude-code-native.md)

### What Native Features Can Replace

**Command blocking (vs. block_commands.py):**
- `permissions.deny` with `Bash(pattern)` rules can block specific command patterns declaratively [1]
- Built-in command blocklist blocks `curl` and `wget` by default [5]
- Built-in command injection detection adds a second layer [5]
- Chain awareness prevents bypass via `&&` [1]

**Path blocking (vs. block_paths.py):**
- `sandbox.filesystem.denyRead` blocks reads at the OS level [1] [39]
- `permissions.deny` with `Read(path)` patterns blocks file reads at the tool level [1]
- OS-level enforcement applies to ALL subprocesses [39]

### What Native Features Cannot Replace

| Capability | Native | This repo's hooks |
|------------|--------|-------------------|
| Regex pattern matching | Glob only [1] | Full regex |
| Complex patterns (e.g., `rm -rf /`) | Cannot express [1] | Supported |
| GWS CLI blocking | Not built-in | 15+ patterns (counted from source) |
| Windows/PowerShell blocking | Not built-in | 12+ patterns (counted from source) |
| Heredoc body stripping | Not built-in | Prevents false positives |
| Custom logic (logging, context) | Not supported | Arbitrary Python |
| Agent escape hatch | `dangerouslyDisableSandbox` [42] | None |

### Recommendation

Use native `permissions.deny` for simple patterns (e.g., `Bash(git push *)`) as a first layer. Keep the repo's hooks for complex regex patterns, GWS CLI blocking, and Windows commands that glob syntax cannot express.

## Dimension 2: Community Claude Code Hooks

Full reference: [reference/community-hooks.md](reference/community-hooks.md)

### Landscape

The community has produced 10+ open-source hook projects for Claude Code safety, ranging from simple shell scripts to SIMD-accelerated Rust pipelines [8]-[19].

### Notable Projects

**dcg (Destructive Command Guard)** [8]: Rust, SIMD-accelerated, <10us for 95%+ commands. Broadest coverage (git, filesystem, database, Kubernetes, cloud providers). Supports 5+ AI agents (Claude Code, Gemini CLI, GitHub Copilot CLI, OpenCode, Aider) [8]. Does not cover GWS CLI or Windows commands.

**claude-code-safety-net** [9]: TypeScript/Bun, semantic argument parsing. Key advantage: distinguishes safe from dangerous command variants (e.g., `git checkout -b` vs `git checkout --`). Claude Code's native deny rules cannot make this distinction.

**Trail of Bits config** [10]: Opinionated defaults with a critical insight: "hooks are not a security boundary — they are structured prompt injection at opportune times for intercepting tool calls and steering behavior."

**nah** [13]: Context-aware safety guard classifying tool calls using contextual rules in milliseconds. Known limitation: `--dangerously-skip-permissions` causes async hook firing, preventing blocking.

### Comparison to This Repo

| Feature | This repo | dcg [8] | safety-net [9] |
|---------|-----------|---------|----------------|
| GWS CLI blocking | Yes | No | No |
| Windows/PowerShell | Yes | Unknown | No |
| Path blocking | Yes (separate hook) | No | No |
| Semantic analysis | No (regex only) | No (regex only) | Yes |
| Performance | Python startup | <10us (Rust/SIMD) | Unknown |
| Multi-agent support | Claude Code | 6+ agents | Claude Code + Copilot |

### Recommendation

Consider adopting ideas from claude-code-safety-net [9] (semantic variant distinction) and nah [13] (supply-chain safety via tighten-only project config). The repo's GWS CLI and Windows coverage is unique — no community project replicates it.

## Dimension 3: AI Agent Guardrails

Full reference: [reference/agent-guardrails.md](reference/agent-guardrails.md)

### Industry Landscape

| Agent | Sandbox Type | Deterministic? | Known Bypasses |
|-------|-------------|----------------|----------------|
| Codex CLI [21] | Kernel (Landlock + seccomp) | Yes | None found |
| Claude Code [39] | OS process (bwrap/Seatbelt) | Yes | Agent self-escape [42] |
| Cursor [25][27] | None (app-level) | No (LLM + string) | 4 CVEs + 2 bypasses |
| Aider [28] | None | N/A | N/A (no restrictions) |
| Amazon Q CLI [32] | None (app-level) | Partially | Enforcement bug [33] |
| Cline [37] | None (app-level) | No (LLM-evaluated) | .clinerules bypass [38] |

**Codex CLI** [21] is the only agent with kernel-enforced, deterministic sandboxing across all three major platforms (Seatbelt/macOS, Landlock+seccomp/Linux, Restricted Token/Windows). It is the gold standard.

**Cursor** [25][27] has the most documented bypasses — multiple independent security researchers have demonstrated that application-layer string matching without OS enforcement is fundamentally unreliable.

**Amazon Q CLI** [33] has a critical enforcement bug: denied commands execute and return output before the permission system blocks them.

### Cross-Agent Frameworks

**IronCurtain** [34]: Semantic interposition on MCP tool calls, English-to-deterministic-rules compilation, V8 isolate execution. Key quote: "Unlike guardrails that rely on the LLM itself, the policy engine operates entirely outside the model."

**LlamaFirewall (Meta)** [35]: Three-component system (PromptGuard 2, Agent Alignment Checks, CodeShield). >90% reduction in attack success on AgentDojo benchmark [35].

### Recommendation

The repo's hooks occupy the same architectural niche as Cursor's allowlist/denylist and Amazon Q's deniedCommands — application-layer, pattern-based blocking. This is adequate for catching honest mistakes but insufficient for adversarial scenarios. For autonomous operation, complement hooks with OS-level sandboxing (following the Codex CLI model).

## Dimension 4: Container/Sandbox Approaches

Full reference: [reference/container-sandbox.md](reference/container-sandbox.md)

### Performance for AI Coding Workloads

| Approach | Overhead | Notes |
|----------|----------|-------|
| Bubblewrap/nsjail | Negligible | Process sandboxing, minimal wrapper |
| Docker container | Low (startup) | Well-understood, broad ecosystem |
| Firecracker microVM | 125ms boot, <5 MiB memory [47] | Full kernel isolation |
| gVisor | <3% CPU-bound [45]; 2.2-216x I/O [46] | Highly workload-dependent |
| PreToolUse hooks (this repo) | Python startup per call | No OS isolation |

### Claude Code's Native Sandbox Already Exists

Claude Code uses bubblewrap (Linux) and Seatbelt (macOS) for OS-level sandboxing [39]. This overlaps significantly with block_paths.py. However, the native sandbox has a known escape hatch: the agent can autonomously disable it when it interferes with task completion [42].

### Industry Direction

The trend among hyperscalers is toward hardware-level isolation for AI agent workloads [47]:
- AWS: Firecracker for Lambda/Fargate
- Google: gVisor for Search/Gmail, Kubernetes Agent Sandbox [55]
- Docker: microVM-based sandboxes for coding agents [41]
- Coder: nsjail for Agent Boundaries [44]

Application-layer hooks are not considered sufficient for untrusted code execution by any major platform.

### Recommendation

The repo's hooks provide defense-in-depth that complements OS-level sandboxing. They catch destructive patterns at the application layer before commands reach the sandbox, work cross-platform, and cannot be disabled by the agent. For production autonomous workflows, add container/VM isolation.

## Dimension 5: Shell-Level Security Tools

Full reference: [reference/shell-security.md](reference/shell-security.md)

### Applicability Assessment

| Tool | Applicable? | Why |
|------|-------------|-----|
| Landlock [68] | **Yes** | Unprivileged, path-based, no root, kernel 5.13+ |
| AppArmor [57] | Partial | Good path control but requires root for profile loading |
| seccomp-bpf [64] | Partial | Cannot filter by file paths — only syscall numbers |
| Capabilities [67] | Minimal | Restricts privileged ops, not file access or commands |
| SELinux [61] | No | Overkill — complexity disproportionate to benefit |
| rbash [65] | No | "Trivially bypassed" — dozens of documented escape methods |

### Landlock Is the Most Relevant

Landlock LSM [68][69] enables unprivileged processes to restrict their own filesystem access — no root, no system-wide configuration. Available since kernel 5.13.

It is already used by:
- Codex CLI for filesystem isolation [21]
- ai-jail for Claude Code sandboxing [56]
- Coder Agent Boundaries (via landjail) [44]

**Key difference from hooks:** Landlock operates at the kernel level. A process that restricts itself via Landlock cannot undo the restriction — monotonically decreasing privileges [68]. PreToolUse hooks operate at the application level and could theoretically be bypassed by a sufficiently creative agent (though Claude Code's hook architecture makes this difficult in practice).

### Defense-in-Depth Model

The established Linux security layering [73]:

| Layer | Mechanism | Analogy to AI Agent |
|-------|-----------|-------------------|
| 1 | Capabilities | Drop unnecessary privileges |
| 2 | seccomp-bpf | Filter dangerous syscalls |
| 3 | AppArmor/SELinux | Restrict file/network access |
| 4 | Landlock | Application-embedded restrictions |
| **5** | **PreToolUse hooks** | **Semantic-level pattern matching** |

The repo's hooks add a **fifth layer** that operates at the semantic level — understanding command intent (e.g., "this is a git push") rather than raw syscall numbers. This layer catches patterns that lower layers cannot express.

### Recommendation

Consider adding Landlock-based filesystem restrictions as a complement to hooks on Linux. The hooks remain valuable for cross-platform coverage, command-level semantics (GWS CLI, Windows commands), and the ability to provide feedback to the agent explaining why a command was blocked.

## Recommendations

### Keep the Hooks

The repo's hooks provide unique value that no alternative fully replicates:
1. **GWS CLI mutation blocking** — no other tool covers this
2. **Windows/PowerShell coverage** — only Codex CLI also addresses this
3. **Cross-platform Python** — works on any platform without kernel support
4. **No agent escape hatch** — unlike the native sandbox
5. **Explanatory feedback** — blocked commands get stderr messages guiding the agent

### Complement with Native Features

Add `permissions.deny` rules for simple patterns that glob syntax can express:
```json
{
  "permissions": {
    "deny": [
      "Bash(git push *)",
      "Bash(sudo *)",
      "Read(~/.ssh/*)",
      "Read(~/.aws/*)"
    ]
  }
}
```

This provides a fast, in-process first check before hooks even fire.

### Consider for Autonomous Workflows

For unattended operation with `--dangerously-skip-permissions`:
1. Enable Claude Code's native sandbox (`sandbox.enabled: true`)
2. Configure `sandbox.filesystem.denyRead` for sensitive paths
3. Run inside a Docker Sandbox or devcontainer for blast radius containment
4. Keep hooks as the semantic layer catching command-level patterns

### Watch These Projects

- **IronCurtain** [34] — semantic interposition with deterministic rules compiled from English
- **nah** [13] — supply-chain safety (tighten-only project config)
- **Landlock** [68] — unprivileged kernel-level filesystem restrictions
- **LlamaFirewall** [35] — chain-of-thought auditing for goal hijacking detection
